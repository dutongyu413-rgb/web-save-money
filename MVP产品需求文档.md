# 反向记账 MVP 产品需求文档

> 文档状态：第二版  
> 产品名称：待定  
> 产品形态：移动端优先的响应式网页  
> 数据方式：全部业务数据仅保存在当前设备的当前浏览器中，支持密码加密备份与恢复，不提供服务器存储或云同步；仅使用 Umami 记录匿名页面访问和关键功能是否成功使用
> 备份文件：使用 `.backup` 扩展名，例如 `反向记账备份_2026-07-16.backup`

## 1. 产品定位

### 1.1 一句话定义

一款帮助用户通过“记录收入、先存下一笔、查看储蓄率”告别月光的极简储蓄记账工具。

### 1.2 目标用户

- 不愿意逐笔记录消费的人。
- 想记账但难以长期坚持的人。
- 收入不低，但经常月光的人。
- 想开始储蓄，却缺少明确行动反馈的人。

### 1.3 用户问题

传统记账要求用户持续记录大量琐碎支出，操作成本高，而且通常只能在钱花掉以后复盘。长期使用时，用户还可能因为补记、漏记或为了账目好看而修改数据，逐渐失去记录价值。

本产品把注意力从“钱花到哪里”转移到“最终留下多少”，让用户只记录少量、高价值的数据。

### 1.4 核心闭环

```text
记录收入
→ 根据目标储蓄率得到建议储蓄金额
→ 用户实际存钱并记录储蓄
→ 查看收入、净储蓄、储蓄率和总支出
```

### 1.5 核心承诺

> 不用记录每笔支出，只记收入和存下的钱。

### 1.6 产品原则

- 收入、储蓄和储蓄率是第一信息层级。
- MVP 完全不提供支出记录入口。
- 总支出由收入和净储蓄自动计算，不要求用户维护支出流水。
- 系统不能自动认定用户已经储蓄，必须由用户主动记录。
- 数据允许出现负数、超过 100% 等真实结果，不为了好看而修正。
- 用户可以逐笔记录，也可以按月汇总记录。
- 不引入真实银行卡余额、复式记账或复杂资金流转概念。

## 2. MVP 范围

### 2.1 MVP 包含

- 收入记录的新增、查看、编辑和删除。
- 储蓄记录的新增、查看、编辑和删除。
- 正数和负数储蓄记录。
- 月度收入、净储蓄、储蓄率和自动计算的总支出。
- 年度收入、净储蓄和年度储蓄率。
- 固定储蓄率目标。
- 记录收入后的建议储蓄引导。
- 每月收入与储蓄柱状图。
- 每月储蓄率折线图。
- 年度收入、储蓄和储蓄率汇总。
- 本地持久化存储。
- 使用密码加密的备份导出与覆盖恢复。
- 移动端优先的响应式布局。

### 2.2 MVP 不包含

- 任何支出记录入口。
- 支出分类、支出明细或固定支出。
- 未记账支出或支出拆解。
- 注册、登录和云同步。
- 银行卡、支付宝、微信账单自动导入。
- 家庭共享或多人账本。
- 收入及储蓄子账户。
- 真实银行卡余额管理。
- 退款、报销、借款、还款和资产转移等专门账务类型。
- 自动识别或自动补记储蓄。
- 消息推送和系统通知。
- 预算管理及消费限额。

## 3. 核心概念

### 3.1 收入账户

系统内置且不可删除的虚拟账户，用于汇总所有收入。MVP 不提供账户备注，也不读取或维护真实银行卡余额。

### 3.2 储蓄账户

系统内置且不可删除的虚拟账户，用于汇总用户主动认定的储蓄。

- 正数表示增加储蓄。
- 负数表示取用过去的储蓄。
- 月度净储蓄可以为零或负数。

MVP 不提供账户备注，也不读取或维护真实资产余额。

### 3.3 总支出

```text
总支出 = 当月总收入 - 当月净储蓄
```

总支出是由收入和储蓄倒推得到的参考数字，而不是支出流水之和。界面名称统一使用“总支出”，不使用“推算总支出”。

首次查看说明：

> 总支出由收入减去净储蓄自动计算。跨月存钱或取用过去的储蓄时，结果可能为负数，仅用于观察整体资金情况。

### 3.4 储蓄率目标

用户设置固定目标储蓄率。默认值为 20%，修改时使用纵向数字滚轮，可选范围为 1% 到 100%，每次变化 1%。目标参与建议储蓄金额和达成状态的计算，但不会自动创建储蓄记录。

## 4. 信息架构

底部一级导航包含：

1. 首页
2. 趋势
3. 设置

MVP 全局仅提供两个核心记录动作：

- 记收入
- 记储蓄

任何一级页面和快捷入口都不出现“记支出”。

## 5. 页面结构

### 5.1 首页

#### 页面目标

让用户立即看到当前自然月赚了多少、留下多少、储蓄率是多少，并快速记录收入或储蓄。

#### 信息顺序

1. 月份切换。
2. 储蓄率及目标达成状态。
3. 当月收入和当月净储蓄。
4. 距离目标还差多少。
5. 总支出。
6. “记收入”和“记储蓄”两个操作。

#### 页面字段

- 当前自然月，默认本月。
- 月份仅允许查看本月及历史月份，不提供未来月份空页面。
- 查看本月时禁用“下个月”；查看历史月份时在日期右侧显示“回到本月”快捷按钮。按钮出现或隐藏时，首页主体元素位置保持不变。
- 首页不设置右上角的设置快捷图标，统一从底部“设置”导航进入。
- 当月储蓄率。
- 目标储蓄率。
- 储蓄率旁的目标编辑按钮。
- 目标达成状态。
- 当月总收入。
- 当月净储蓄。
- 建议储蓄金额。
- 距离目标储蓄金额。
- 总支出。

#### 交互要求

- 支持切换上月和下月。
- 收入、储蓄和储蓄率的视觉层级必须高于总支出。
- 首页不展示支出明细、分类或拆解暗示。
- 点击收入进入收入记录列表。
- 点击储蓄进入储蓄记录列表。
- 首次点击总支出时显示计算说明；之后仍可通过信息图标查看。
- 负数或超过 100% 的结果不得隐藏。
- 点击储蓄率旁的编辑图标，可以直接修改目标储蓄率，无需进入设置页。

#### 快捷操作

- “记收入”为次要按钮。
- “记储蓄”为主要按钮。
- 不设置“记支出”按钮。

### 5.2 记录收入后的储蓄引导

收入保存成功后，显示轻量确认页或底部弹层：

```text
收入已记录 ¥10,000
按照 20% 的目标，建议存下 ¥2,000

[记下这笔储蓄]
[稍后再说]
```

交互规则：

- 建议金额基于用户本月累计收入和目标储蓄率，而不是只根据本次收入。
- 显示本月目标储蓄总额、当前已储蓄和仍需储蓄。
- 点击“记下这笔储蓄”进入新增储蓄页，并预填仍需储蓄金额。
- 点击“稍后再说”返回首页，不创建储蓄记录。
- 引导不能声称用户已经完成真实转账。
- 当本月目标已经达成时，显示“本月储蓄目标已达成”，不要求继续储蓄。

### 5.3 收入记录列表

- 显示当前月份收入合计。
- 按发生日期倒序排列记录。
- 每条记录显示记录类型、日期、金额和备注摘要。
- 支持切换月份。
- 支持新增、编辑和删除。
- 新增入口只保留页面底部的“记一笔收入”主按钮，顶部不设置重复的加号入口。
- 删除前二次确认。

### 5.4 新增/编辑收入

字段：

- 金额：必填，必须大于 0，最多两位小数。
- 日期：必填，默认当天，不得晚于当前本地日期。
- 备注：选填，最长 200 个字符。

保存后按照发生日期归入对应自然月，并立即更新汇总和目标差额。

### 5.5 储蓄记录列表

- 显示当前月份净储蓄合计。
- 按发生日期倒序排列记录。
- 正数和负数储蓄清晰区分，但负数不作为错误展示。
- 支持切换月份。
- 支持新增、编辑和删除。
- 新增入口只保留页面底部的“记一笔储蓄”主按钮，顶部不设置重复的加号入口。
- 删除前二次确认。

### 5.6 新增/编辑储蓄

为了避免要求用户输入负号，先选择操作类型：

- “增加储蓄”：保存为正数。
- “取用储蓄”：保存为负数。

字段：

- 操作类型：必填，默认“增加储蓄”。
- 金额：必填；界面输入正数，系统根据类型保存正负号。
- 日期：必填，默认当天，不得晚于当前本地日期。
- 备注：选填，最长 200 个字符。

选择“取用储蓄”时显示：

> 这笔金额会减少本月净储蓄，储蓄率可能为负数。

系统不校验可用储蓄余额，因为 MVP 不维护真实资产余额。

### 5.7 趋势页

MVP 只展示：

1. 每月收入与净储蓄柱状图。
2. 每月储蓄率折线图。
3. 当前年度总收入、年度净储蓄和年度储蓄率。

交互要求：

- 默认当前自然年，支持切换年份。
- 点击图表数据点显示当月具体值。
- “每月收入与净储蓄”图显示金额纵轴，并使用人民币缩略刻度。
- “每月储蓄率”图显示百分比纵轴，纵轴范围根据真实数据动态扩展。
- 收入为零的月份不绘制储蓄率数值点，显示“-”。
- 负数或超过 100% 时不能截断真实数据。

### 5.8 设置页

包含：

- 导出加密备份。
- 从备份恢复。
- 页面顶部的本地数据提醒。
- 清空全部数据。
- 产品版本号。

“清空全部数据”在设置页使用一行红色文字展示，不设置“危险操作”标题、警示卡片或背景色。点击后必须二次确认，并明确提示将永久删除当前浏览器中的全部收入、储蓄和设置且无法撤销。确认清空后，收入和储蓄记录归零，目标储蓄率恢复为默认值 20%。取消操作不得修改任何数据。

设置页顶部固定显示一句提醒，不设置独立二级页面：

> 数据保存在当前浏览器中，建议定期导出加密备份。

### 5.9 备份与恢复

#### 导出加密备份

1. 输入备份密码并再次确认。
2. 密码至少 8 个字符。
3. 两次输入必须一致。
4. 密码输入框字号不得小于 16px，避免 iOS 聚焦时自动放大页面。
5. 生成成功前不展示文件名、记录数量或文件预览卡片。
6. 系统生成包含全部业务数据、设置、数据版本和导出时间的加密文件。
7. 备份文件由用户自行保存，可用于当前设备或其他设备恢复。
8. 产品不保存备份文件和密码，也不提供密码找回能力。
9. 文件扩展名固定为 `.backup`；该文件为应用专用加密备份，不能用 Excel 直接打开。

#### 覆盖恢复

1. 用户选择备份文件。
2. 系统验证文件格式。
3. 用户输入备份密码。
4. 系统完成解密和完整性校验。
5. 明确提示恢复会覆盖当前全部数据。
6. 用户二次确认后执行覆盖。
7. 恢复完成后重新计算全部汇总并返回首页。

恢复不提供数据合并。密码错误、文件损坏或版本不兼容时，不得修改当前数据。恢复成功后，备份中的数据写入执行恢复操作的当前浏览器，不会自动同步回其他设备或浏览器。

### 5.10 本地保存说明

- 收入、储蓄、设置和其他业务数据全部保存在当前浏览器的本地数据库中。
- 刷新页面、正常关闭浏览器或重启设备，通常不会删除数据。
- 同一链接在不同设备或不同浏览器中打开时，各自拥有独立数据，不会自动同步。
- 浏览器本地数据通常与网站来源绑定；协议、域名或端口发生变化时，可能无法访问原地址下的数据。
- 产品不把用户业务数据上传到服务器，也无法在服务端帮助用户找回数据；匿名使用统计不包含金额、日期、备注、目标数值、密码和备份内容。
- 用户应定期导出加密备份，并自行妥善保存备份文件和密码。

可能导致数据消失或无法访问的情况包括：

- 用户主动清除该网站的浏览器数据。
- 浏览器或操作系统因存储空间、权限或清理策略删除网站数据。
- 使用隐私或无痕模式，关闭相关窗口后数据被删除。
- 卸载浏览器、重装系统、恢复设备或更换设备。
- 更换浏览器、浏览器用户配置、网站域名、协议或端口。
- 浏览器本地数据库损坏。
- 产品升级的数据迁移发生不可恢复的异常。

## 6. 数据模型

所有金额以“分”为单位存储为整数，禁止使用浮点数直接持久化。所有主键使用不可预测的唯一字符串。

### 6.1 AppSettings

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| id | string | 是 | 固定设置记录 ID |
| schemaVersion | integer | 是 | 本地数据结构版本 |
| savingsRateTarget | number | 是 | 小数形式，如 0.2 表示 20% |
| currency | string | 是 | MVP 固定为 CNY |
| createdAt | datetime | 是 | 创建时间 |
| updatedAt | datetime | 是 | 更新时间 |

储蓄率目标允许设置为 0% 至 100%。建议新用户初始目标为 10%，在首次使用流程中允许修改；不强迫用户接受默认值。

### 6.2 IncomeRecord

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| id | string | 是 | 唯一 ID |
| amountCent | integer | 是 | 正整数金额，单位分 |
| occurredOn | date | 是 | 发生日期，本地日期 |
| note | string | 否 | 备注 |
| createdAt | datetime | 是 | 创建时间 |
| updatedAt | datetime | 是 | 更新时间 |

### 6.3 SavingsRecord

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| id | string | 是 | 唯一 ID |
| amountCent | integer | 是 | 非零整数；正数为增加，负数为取用 |
| occurredOn | date | 是 | 发生日期，本地日期 |
| note | string | 否 | 备注 |
| createdAt | datetime | 是 | 创建时间 |
| updatedAt | datetime | 是 | 更新时间 |

### 6.4 BackupEnvelope

备份文件逻辑结构至少包含：

- 文件格式标识。
- 备份格式版本。
- 加密参数。
- 密文数据。
- 完整性校验信息。
- 导出时间。

明文业务数据仅包含 AppSettings、IncomeRecord 和 SavingsRecord 的完整集合。

## 7. 计算规则

### 7.1 月度归属

所有记录按照 `occurredOn` 的本地自然日期归入自然月，与记录创建时间无关。修改发生日期后，原月份和新月份的汇总都要立即更新。

### 7.2 当月总收入

```text
当月总收入 = 当月所有收入记录金额之和
```

### 7.3 当月净储蓄

```text
当月净储蓄 = 当月所有储蓄记录金额之和
```

正负记录直接相加。

### 7.4 当月储蓄率

当月总收入大于 0 时：

```text
当月储蓄率 = 当月净储蓄 ÷ 当月总收入 × 100%
```

当月总收入等于 0 时，储蓄率不可计算，显示“-”。储蓄率允许小于 0% 或大于 100%。默认显示一位小数，整数可省略小数位。

### 7.5 目标储蓄金额

```text
目标储蓄金额 = 当月总收入 × 目标储蓄率
```

结果按“分”四舍五入。

### 7.6 距离目标金额

```text
距离目标金额 = 目标储蓄金额 - 当月净储蓄
```

- 结果大于 0：显示“距离目标还差 ¥X”。
- 结果等于 0：显示“刚好达成目标”。
- 结果小于 0：显示“已超过目标 ¥X”。
- 收入为 0：不显示目标差额。

### 7.7 目标达成状态

当月收入大于 0 时：

```text
当月净储蓄 >= 目标储蓄金额：已达成
当月净储蓄 < 目标储蓄金额：未达成
```

收入为 0 时显示“暂无可比较的收入”。

### 7.8 总支出

```text
总支出 = 当月总收入 - 当月净储蓄
```

总支出允许为零或负数。系统不得自动调整记录以改变结果。

### 7.9 年度指标

```text
年度总收入 = 该自然年所有收入记录金额之和
年度净储蓄 = 该自然年所有储蓄记录金额之和
年度储蓄率 = 年度净储蓄 ÷ 年度总收入 × 100%
```

年度总收入为 0 时，年度储蓄率显示“-”。年度储蓄率必须用年度总额直接计算，不能取各月储蓄率的算术平均值。

### 7.10 显示规则

- 默认人民币符号为 `¥`。
- 金额使用千位分隔符。
- 整元不显示小数；存在角分时显示两位小数。
- 负金额显示为 `-¥1,000`。
- 不使用红色把负储蓄强制表达为错误。

## 8. 异常状态

### 8.1 无数据

- 收入、储蓄和总支出显示 `¥0`。
- 储蓄率显示“-”。
- 主操作为“记第一笔收入”。
- 不显示储蓄遗漏提示。

### 8.2 有收入但没有储蓄记录

- 净储蓄为 `¥0`。
- 储蓄率为 `0%`。
- 总支出等于收入，即“没存下来的部分视为已经花掉”。
- 显示轻提示：“本月还没有储蓄记录。如果确实没有存下钱，可以忽略。”
- 提示可关闭，不阻止使用。

### 8.3 储蓄记录相抵为零

例如 `+¥1,000` 和 `-¥1,000`：

- 净储蓄显示 `¥0`。
- 不显示“没有储蓄记录”提示。
- 两笔明细都保留。

### 8.4 收入为零但存在储蓄

- 如实显示收入和净储蓄。
- 储蓄率显示“-”。
- 目标状态显示“暂无可比较的收入”。
- 总支出仍按公式计算，允许为负数。

### 8.5 净储蓄为负数

- 如实显示负数净储蓄和负数储蓄率。
- 显示解释：“本月取用的储蓄多于新增储蓄。”
- 不阻止保存。

### 8.6 储蓄率超过 100%

- 如实显示实际百分比。
- 目标状态为已达成。
- 可提示：“本月储蓄可能包含过去结余，因此可能超过本月收入。”

### 8.7 总支出为负数

- 如实显示负数。
- 显示说明：“总支出由收入减去净储蓄得出。跨月资金可能产生负数，仅供参考。”
- 不自动修改数据。

### 8.8 非法金额

- 空值、零值、非数字或超过两位小数时禁止保存并就地提示。
- 收入不接受负数。
- 储蓄金额输入框只接受正数，由操作类型决定保存正负号。
- 单笔金额建议上限为 9,999,999,999.99 元。

### 8.9 本地存储不可用或写入失败

- 启动时检测持久化能力。
- 无法持久化时不得让用户误以为数据已经保存。
- 保存失败时保留表单内容并提示重试。
- 检测到隐私/无痕模式、存储配额不足或浏览器限制时，明确说明数据可能在关闭页面后消失。
- 建议用户更换正常浏览模式、释放存储空间或及时导出备份。

### 8.10 备份异常

- 密码错误、文件损坏、格式不支持或版本不兼容时终止恢复。
- 异常时不得修改当前数据。
- 覆盖恢复采用事务式流程：完整解密、校验并准备成功后，再一次性替换当前数据。

### 8.11 日期与时区

- 发生日期保存为不带时区的 `YYYY-MM-DD`。
- 设备时区变化不能使记录移动到前一天或后一天。

## 9. 本地数据与安全

- 所有业务数据仅保存在当前设备、当前浏览器、当前网站来源对应的本地存储中。
- 优先使用 IndexedDB 保存业务数据，并在条件允许时申请持久化存储，降低浏览器自动回收数据的概率；该申请不构成数据永不丢失的保证。
- 数据不在不同设备、浏览器、浏览器用户配置或网站来源之间自动共享。
- 不通过 URL、日志或分析事件上传金额、日期和备注。
- 匿名统计事件统一使用 `save_money_` 前缀，只允许页面分类、创建/编辑、增加/取用等非敏感属性。
- 不在服务器保存业务数据、备份文件或备份密码。
- 页面不得声称数据永不丢失。
- 备份使用成熟的带认证加密方案，优先使用浏览器原生 Web Crypto。
- 密码派生包含随机盐，每次加密使用随机初始化参数。
- 不保存用户备份密码或可还原密码的内容。
- 产品升级使用 `schemaVersion` 执行经过验证的数据迁移。

## 9.1 视觉配色

- 主题色使用可信、清晰的蓝色，用于导航选中、主要数据、链接、输入焦点和常规主操作。
- 辅助色使用温暖橙色，用于“记储蓄”、储蓄目标、目标达成和少量需要强调的行动反馈。
- 橙色不用于大面积背景，不与蓝色平均分配；收入与通用功能优先使用蓝色，储蓄行为优先使用橙色。
- 错误状态使用独立的低饱和红色，不能用橙色代替错误含义。
- 背景以冷白和浅蓝灰为主，正文使用深蓝黑，确保金额与按钮满足可读性对比度要求。

## 10. 非功能要求

### 10.1 兼容性

- 手机浏览器优先，支持手机竖屏和桌面浏览器。
- 交互控件满足触屏尺寸要求。
- 后续可嵌入小程序子页面，但 MVP 不依赖小程序 API。

### 10.2 性能

- 常规本地数据量下，首页汇总即时呈现。
- 新增、编辑或删除后，相关指标立即更新。
- 图表不能阻塞核心记录操作。

### 10.3 可用性

- 新用户无需理解会计术语即可完成首笔收入和储蓄记录。
- 使用“增加储蓄/取用储蓄”代替手动输入正负号。
- 储蓄建议是行动引导，不得伪装成自动转账。
- 所有破坏性操作明确说明影响。

## 11. MVP 验收标准

### 11.1 核心闭环

给定：

- 当月收入 `¥10,000`
- 目标储蓄率 `30%`
- 增加储蓄 `¥3,000`

首页显示：

- 收入 `¥10,000`
- 净储蓄 `¥3,000`
- 储蓄率 `30%`
- 目标状态“已达成”
- 总支出 `¥7,000`

再新增一笔“取用储蓄” `¥1,000` 后显示：

- 净储蓄 `¥2,000`
- 储蓄率 `20%`
- 距离目标还差 `¥1,000`
- 总支出 `¥8,000`

### 11.2 零储蓄

收入为 `¥10,000` 且没有储蓄记录时：

- 储蓄显示 `¥0`。
- 储蓄率显示 `0%`。
- 总支出显示 `¥10,000`。
- 显示非阻断储蓄提示。

### 11.3 年度口径

年度储蓄率使用年度总收入和年度净储蓄直接计算，不取月度储蓄率平均值。

### 11.4 本地保存与备份

- 刷新页面和重新打开浏览器后数据仍存在。
- 关闭浏览器和重启设备后，在正常浏览模式且网站数据未被清理的情况下，数据仍可访问。
- 不同设备、浏览器、浏览器用户配置或网站来源之间的数据互不自动同步。
- 本地写入失败时不得显示保存成功。
- 正确密码可恢复全部数据。
- 错误密码或损坏文件不能覆盖当前数据。
- 覆盖恢复前必须二次确认。
- 恢复成功后只写入执行恢复操作的当前浏览器。

### 11.5 范围约束

- 所有 MVP 页面均不存在“记支出”按钮。
- 不存在支出分类、支出记录或固定支出数据结构。
- 总支出只能由收入减去净储蓄计算得出。

## 12. 后续独立功能线：支出拆解

支出记录不属于 MVP，但产品不永久否定这项能力。当真实用户明确提出需求，或产品验证需要更深入的资金分析时，可单独立项。

候选形态：

1. 在总支出详情中增加“拆解总支出”二级入口。
2. 在底部增加独立“支出”标签。
3. 支持少量固定大项，而非要求逐笔消费记账。

可能包含：

- 自定义支出分类。
- 支出记录。
- 固定周期支出。
- 已拆解支出和其他支出。

后续即使加入，也应遵守：

- 首页核心仍然是收入、储蓄和储蓄率。
- 默认使用路径不要求记录支出。
- 不使用“未记账”制造用户负担。
- 支出功能是可选拆解能力，不改变“先存后花”的产品主张。

是否启动该功能线，以用户访谈、反馈数量、使用行为和留存数据为依据，不在 MVP 阶段提前实现。
