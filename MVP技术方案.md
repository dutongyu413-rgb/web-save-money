# 反向记账 MVP 技术方案

## 1. 文档目的

本文档用于指导“反向记账”正式 MVP 的开发、测试和上线。早期静态交互原型已经归档，当前仓库以 React + TypeScript 工程作为唯一正式实现。

正式 MVP 的目标是：

- 不建设业务后端和云端数据库。
- 所有收入、储蓄和设置默认只保存在当前浏览器。
- 使用 Umami 记录匿名访问和关键行为，但不传输任何业务数据。
- 支持加密导出 `.backup` 文件，并可在其他设备恢复。
- 保证金额计算、月份归属和数据迁移可测试。
- 保留后续挂载到微信小程序 WebView 的可能性。

## 2. 技术结论

### 2.1 是否需要服务器

不需要业务服务器，也不需要用户系统、登录系统或云端数据库。

但产品作为网页对外访问时，仍需要一个 HTTPS 静态托管地址，用于提供 HTML、CSS、JavaScript 和图标资源。静态托管不接收用户的记账数据。

```text
静态托管/CDN
  仅提供应用代码
        ↓
用户浏览器
  IndexedDB 保存业务数据
  Web Crypto 生成加密备份
        └── Umami：仅发送匿名页面分类和功能成功事件
```

### 2.2 推荐技术栈

| 层级 | 选型 | 用途 |
| --- | --- | --- |
| 构建工具 | Vite | 本地开发、构建和静态发布 |
| 语言 | TypeScript | 数据模型、计算规则和备份结构的类型约束 |
| UI | React | 页面和组件开发 |
| 路由 | React Router | 首页、趋势、设置和二级页面导航 |
| 本地数据库 | IndexedDB + Dexie | 保存收入、储蓄、设置和数据库版本 |
| 匿名统计 | Umami | 页面访问量和关键功能成功事件，不读取业务数据 |
| 数据校验 | Zod | 表单、数据库读取和备份恢复校验 |
| 图标 | Phosphor Icons | 统一线性图标体系 |
| 单元测试 | Vitest | 计算、数据迁移、备份格式测试 |
| 组件测试 | Testing Library | 表单和状态交互测试 |
| 端到端测试 | Playwright | 完整用户流程和浏览器兼容测试 |

MVP 不引入全局状态库。页面状态使用 React 状态管理，业务数据通过 Repository 和查询 Hook 读取。确实出现跨层状态复杂度后，再评估 Zustand。

## 3. 系统架构

```text
页面与组件
  ↓
应用用例
  记录收入 / 记录储蓄 / 修改目标 / 清空数据 / 备份恢复
  ↓
领域层
  数据模型 / 金额计算 / 月份聚合 / 校验规则
  ↓
数据访问层
  IndexedDB Repository / 数据迁移 / 事务
  ↓
浏览器能力
  IndexedDB / Web Crypto / File API / Blob
```

### 3.1 分层原则

- UI 不直接读写 IndexedDB。
- 所有金额和储蓄率计算集中在纯函数中。
- 备份恢复必须先完整校验，再开启覆盖事务。
- 页面不能保存派生汇总，汇总应由原始记录实时计算。
- 本地日期以用户设备时区为准，不使用 UTC 日期切分自然月。

## 5. 数据存储

### 5.1 为什么正式版使用 IndexedDB

当前原型使用 `localStorage`，适合演示，但正式 MVP 建议使用 IndexedDB：

- 支持结构化记录和索引查询。
- 支持事务，覆盖恢复时更安全。
- 容量通常大于 localStorage。
- 支持数据库版本和迁移。
- 不需要每次修改都重写全部 JSON。

### 5.2 数据库表

#### settings

单例设置表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | `"app"` | 固定主键 |
| targetSavingsRate | number | 界面允许设置 1 到 100；读取旧备份时兼容历史值 |
| currency | `"CNY"` | MVP 固定人民币 |
| createdAt | ISO datetime | 创建时间 |
| updatedAt | ISO datetime | 更新时间 |

#### incomes

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | UUID |
| amountCents | integer | 大于 0 |
| occurredOn | `YYYY-MM-DD` | 本地自然日期 |
| note | string | 最长 200 字符 |
| createdAt | ISO datetime | 创建时间 |
| updatedAt | ISO datetime | 更新时间 |

索引：`occurredOn`。

#### savings

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | UUID |
| amountCents | integer | 增加为正数，取用为负数，不得为 0 |
| occurredOn | `YYYY-MM-DD` | 本地自然日期 |
| note | string | 最长 200 字符 |
| createdAt | ISO datetime | 创建时间 |
| updatedAt | ISO datetime | 更新时间 |

索引：`occurredOn`。

### 5.3 数据库版本

首个正式版本定义为数据库 schema version 1。每次表结构或字段语义变化必须新增迁移，不允许直接覆盖旧结构。

迁移执行失败时：

- 不删除原数据库。
- 停止进入主界面。
- 显示恢复提示和导入备份入口。
- 记录本地错误信息，但不上传用户业务数据。

## 6. 金额与计算

### 6.1 金额

- 持久化金额统一使用整数分。
- 表单输入先按字符串校验，再转换为分。
- 禁止用浮点数直接累计金额。
- 页面格式化与业务计算分离。

### 6.2 月度计算

```text
当月收入 = 当月收入记录金额之和
当月净储蓄 = 当月储蓄记录金额之和
总支出 = 当月收入 - 当月净储蓄
储蓄率 = 当月净储蓄 / 当月收入 × 100%
目标储蓄额 = 当月收入 × 目标储蓄率
```

收入为 0 时，储蓄率返回 `null`，界面显示 `-`，不得显示 `0%`。

总支出、净储蓄和储蓄率允许为负数。储蓄率允许超过 100%。

### 6.3 日期

- 使用 `YYYY-MM-DD` 保存发生日期。
- 月份归属由字符串前 7 位计算。
- 默认日期使用用户设备的本地日期。
- 不允许新增晚于今天的记录。
- 月份浏览最多到当前自然月。

## 7. 备份与恢复

### 7.1 备份文件

文件扩展名固定为 `.backup`，内容为 JSON 加密信封：

```json
{
  "format": "reverse-accounting-backup",
  "version": 1,
  "exportedAt": "ISO datetime",
  "kdf": {
    "name": "PBKDF2",
    "hash": "SHA-256",
    "iterations": 210000,
    "salt": "base64"
  },
  "cipher": {
    "name": "AES-GCM",
    "iv": "base64",
    "ciphertext": "base64"
  }
}
```

### 7.2 加密规则

- 密码至少 8 个字符。
- 密码只在当前操作内存中使用，不保存。
- 每次导出生成新的随机 salt 和 iv。
- 使用 Web Crypto API。
- 密文使用 AES-GCM，恢复时同时校验完整性。
- 文件名不包含收入、储蓄或余额信息。

### 7.3 恢复流程

1. 选择 `.backup` 文件。
2. 检查文件大小和 JSON 外层结构。
3. 输入密码并解密。
4. 使用 Zod 校验全部业务数据。
5. 显示记录数量和覆盖确认。
6. 在一个 IndexedDB 事务中清空旧数据并写入新数据。
7. 事务成功后刷新查询并返回首页。

任何一步失败都不得修改当前数据。

### 7.4 清空全部数据

- 必须二次确认。
- 使用一个 IndexedDB 事务清空收入、储蓄和设置。
- 事务完成后重新创建默认设置，目标储蓄率为 20%。
- 清空失败时保留原数据，并显示失败提示。

## 8. 本地数据安全边界

MVP 的本地数据不做静态加密。原因是网页每次打开都需要自动读取数据，如果没有用户登录或主密码，静态加密无法提供真实保护。

需要向用户明确：

- 数据不会主动上传。
- 同一设备上可以访问该浏览器配置的人，可能访问本地数据。
- 浏览器扩展、恶意脚本或设备失窃不在本地存储的保护范围内。
- 加密只用于用户主动导出的备份文件。

## 9. UI 状态与错误处理

每个关键流程必须覆盖以下状态：

| 场景 | 必备状态 |
| --- | --- |
| 首页 | 有数据、无收入、负储蓄、超过 100%、历史月份 |
| 记录列表 | 有记录、空列表、编辑、删除确认 |
| 表单 | 初始、校验失败、保存中、保存失败、保存成功 |
| 趋势 | 全年无数据、部分月份、负数、极端值 |
| 导出 | 密码不一致、生成中、生成失败、成功 |
| 恢复 | 文件错误、密码错误、版本不兼容、覆盖确认、事务失败、成功 |
| 清空 | 二次确认、执行失败、完成 |

错误信息使用用户可理解的语言，不展示堆栈、数据库异常或加密内部细节。

## 10. 可访问性

- 所有按钮可通过键盘操作。
- 图标按钮必须有 `aria-label`。
- 表单使用可见 label，不以 placeholder 代替。
- 错误信息与对应字段关联。
- 颜色不是唯一的信息表达方式。
- 图表提供可访问名称和文字摘要。
- 动效遵守 `prefers-reduced-motion`。
- 正文和按钮达到 WCAG AA 对比度。

## 11. 测试方案

### 11.1 单元测试

重点覆盖：

- 元转分和分转展示。
- 月份归属和本地日期。
- 收入、净储蓄、总支出、储蓄率和目标差额。
- 0 收入、负储蓄、负总支出、超过 100%。
- 备份加密和解密。
- 密码错误、密文损坏和版本不兼容。
- 数据库迁移。

### 11.2 端到端测试

核心流程：

1. 首次打开应用。
2. 记录收入并接受或跳过储蓄提示。
3. 增加储蓄和取用储蓄。
4. 编辑、删除和跨月查看。
5. 修改目标储蓄率。
6. 导出备份。
7. 恢复备份并验证汇总一致。
8. 取消清空与确认清空。
9. 刷新和重启浏览器后数据仍存在。

### 11.3 浏览器范围

首版至少验证当前主流版本的：

- iOS Safari。
- Android Chrome。
- 桌面 Chrome。
- 微信内置浏览器和小程序 WebView，在计划接入前单独验收。

## 12. 构建与发布

### 12.1 环境

```text
development  本地开发
preview      内部测试
production   正式地址
```

不同环境必须使用不同域名或路径，并明确提示浏览器本地数据按网站来源隔离。测试地址的数据不会自动出现在正式地址。

### 12.2 静态发布

MVP 使用 GitHub Pages 发布，正式地址为：

```text
https://dutongyu413-rgb.github.io/web-save-money/
```

推送到 `main` 分支后，GitHub Actions 自动运行测试并执行 `npm run build:pages`，只把生成的 `dist/` 发布到 Pages。`dist/` 不提交到 Git 仓库。Pages 构建使用 `/web-save-money/` 资源基础路径，本地开发仍使用 `/`，避免影响本地预览地址。

构建产物仅包含静态资源，后续也可迁移到对象存储、CDN 或其他静态网站托管服务。

上线要求：

- HTTPS。
- 固定正式域名。
- 正确的缓存版本策略。
- Umami 统计脚本不得读取业务数据，事件属性只允许固定的非敏感分类。
- 配置基础 CSP 和安全响应头。

### 12.3 匿名统计

- 生产构建使用 `web-save-money` 对应的 Umami Website ID；本地开发默认停用统计，可通过 `VITE_UMAMI_WEBSITE_ID` 显式开启联调。
- 自定义事件统一使用 `save_money_` 前缀，与同一账号下的其他项目区分。
- 收入、储蓄、目标、备份等事件只在业务操作真正成功后上报。
- 只允许页面名称、创建/编辑、增加/取用等固定分类，禁止发送金额、日期、备注、目标数值、记录 ID、密码、文件名、记录数量和备份内容。
- 基础页面统计排除 Hash 路由内容，避免编辑页面的记录 ID 出现在访问 URL 中。
- Umami 未加载、被拦截或上报异常时必须静默失败，不得影响任何记账功能。

### 12.4 小程序接入

MVP 先发布为移动端 Web 应用。后续小程序仅通过 WebView 打开正式 HTTPS 地址，不复制业务数据到小程序本地存储。

正式接入前需要单独验证：

- WebView 域名配置。
- 文件下载和上传能力。
- IndexedDB 在目标 WebView 中的持久性。
- 小程序升级或微信清理缓存后的数据表现。
- 平台当时的上架和隐私规则。

## 13. 性能目标

- 首屏不依赖网络 API。
- 首次加载资源尽量控制在 300 KB gzip 以内，不含字体。
- 关键点击到界面反馈低于 100 ms。
- 1000 条收入和 1000 条储蓄记录下，首页月度计算保持流畅。
- 图表只聚合当前年份数据。

## 14. 开发阶段

### 基础工程

- 初始化 Vite、React 和 TypeScript。
- 建立设计令牌、路由、错误边界和测试环境。
- 建立 IndexedDB schema 和 Repository。

### 核心记账

- 收入和储蓄的新增、编辑、删除。
- 月份切换、首页计算和目标储蓄率。
- 空状态、错误状态和二次确认。

### 趋势与数据安全

- 年度趋势图。
- 加密导出和覆盖恢复。
- 清空全部数据。
- 数据迁移和异常恢复。

### 上线准备

- 移动端兼容测试。
- 可访问性检查。
- 静态部署和缓存策略。
- 小范围真实用户试用。

## 15. MVP 验收标准

- 刷新、关闭浏览器和重启设备后数据仍可读取。
- 所有计算规则与产品文档一致。
- 不存在未来月份浏览和未来日期记录。
- 备份可在空白浏览器环境中成功恢复。
- 密码错误或文件损坏不会覆盖当前数据。
- 清空操作必须二次确认且可以取消。
- 无任何业务数据请求发送到服务器。
- Umami 事件名称均带 `save_money_` 前缀，且事件属性不包含任何业务数据。
- 首页、列表、趋势、设置和备份流程在手机宽度下无溢出。

## 16. 暂不进入 MVP

- 登录和账号体系。
- 云同步。
- 银行或支付平台自动导入。
- 支出逐笔记录。
- 多币种。
- 多收入账户和多储蓄账户。
- 消息推送。
- 原生小程序重写。
